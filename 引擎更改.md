# 引擎更改

## 1. 文档目的

本文档用于记录当前这一轮对地图引擎的实际改动，重点说明：

- 已经落地了哪些能力。
- 对外新增了哪些接口与信号。
- 内部结构做了哪些拆分。
- 当前实现的边界在哪里。
- 下一步应该继续推进什么。

本文档描述的是“已经完成的改动”，不是长期规划，也不是 P0 全量完成说明。

## 2. 本轮改动范围

本轮改动覆盖 P0 的 LayerRuntime 前两步，以及 StyleRuntime 的第一步拆分，没有进入 EditRuntime。

本轮已完成的内容包括：

- 新增 LayerManager，作为图层与数据源管理的内部统一入口。
- 让 Map 的部分 source/layer 接口改为委托给 LayerManager。
- 为图层管理补充显隐、排序、分组和元数据读取能力。
- 为图层管理补充批量更新入口。
- 为图层与数据源管理补充事件通知通道。
- 新增 StyleManager，作为 layout/paint 更新与样式补丁提交的内部统一入口。
- 让 Map 的样式属性接口改为委托给 StyleManager。
- 为样式管理补充样式补丁提交入口与批量事件合并入口。
- 为样式管理补充属性白名单约束与错误状态输出。
- 为样式管理补充样式快照读取与简单表达式入口。
- 为样式管理补充 layer 级状态样式入口与基线样式恢复能力。
- 新增 StyleExpression builder，统一常用表达式片段构造方式。
- 为样式变更补充事件通知通道。
- 更新能力规划文档，补齐 P0 拆分说明与执行边界。

本轮未完成的内容包括：

- EditRuntime 的编辑会话模型。
- LayerRuntime 的完整事务回滚能力。
- StyleRuntime 的完整 feature-state 运行时模型与更高层表达式 builder。
- QueryRuntime、LabelRuntime、ObjectBindingRuntime 等 P1 能力。

## 3. 对外新增能力

### 3.1 图层显隐控制

引擎新增了按图层粒度控制可见性的能力。

新增接口：

- setLayerVisible
- isLayerVisible

能力说明：

- 可以直接设置某个图层为显示或隐藏。
- 当前可见状态会同步到内部图层元数据中。
- 图层不存在时会返回失败并输出警告信息。

### 3.2 图层排序控制

引擎新增了图层顺序调整能力。

新增接口：

- moveLayer

能力说明：

- 支持把一个已有图层移动到另一个图层之前。
- 内部实现不是简单记录顺序，而是通过移除后重新插入的方式完成排序。
- 排序完成后会同步更新内部顺序元数据。

### 3.3 图层分组与组级渲染控制

引擎新增了图层分组元数据管理及按组层级的计算与操作能力。

新增接口：

- setLayerGroup
- groupLayerIds
- setGroupVisible（新增组级操作）
- removeGroup（解除分组绑定）

能力说明：

- 可以为图层附加 groupId。
- 可以按 groupId 查询该组下的图层列表。
- 支持基于 `Group` 宏观调控，修改组显隐或批量解除分组绑定，底层自动使用批量封装（Batch）执行防抖。
- **组状态渲染覆盖机制**：引入了业务侧期望可见性 (`requestedVisibility`) 与受制于层级树的实际渲染可见性 (`actualVisibility`)。当父组（Group）处于隐藏状态时，自动屏蔽所有派生图层的可见性；屏蔽状态下独立的 `setLayerVisible(true)` 会被缓存至期望可见性中，待组可见后恢复。
- `layerVisibilityChanged` 仅在实际渲染可见状态发生变化时发出，不再把“仅期望状态变化”误报成可见性变更。
- 当前分组能力属于运行时元数据能力，避免了向底层引擎污染多余的 `mbgl::style::Layer` 实体树。

### 3.4 图层元数据快照能力

引擎新增了图层元数据读取与快照输出能力。

新增接口：

- layerMetadata
- layerMetadataSnapshot

当前输出的核心字段包括：

- id
- type
- sourceId
- groupId
- visible
- order
- minZoom
- maxZoom

能力价值：

- 为后续诊断、图层控制面板、业务图层树提供基础数据来源。
- 为后续 LayerRuntime、StyleRuntime、QueryRuntime 之间共享统一图层状态打基础。

### 3.5 批量更新与异常防污染入口

引擎新增了图层批量操作与附带异常检查的样式合并入口。

新增接口：

- beginLayerBatch / commitLayerBatch / isLayerBatchActive
- beginStyleBatch / commitStyleBatch / isStyleBatchActive

当前语义说明：

- 批量更新期间操作会立即向引擎下发执行，但事件被挂起不会立刻逐条向外抛出（防止大规模渲染导致的 UI 卡顿）。
- 在 commit 时会统一 flush 事件并发出批量完成信号。
- **快照事务备忘录 (Memento 回滚) **：针对 `StyleBatch` 更新了完整的防污染撤回功能。如果向字典写入不符合预期的属性名或是非法类型而触发 `StyleUnsupportedPropertyError`，`commitStyleBatch` 会激活基于当前 layer 快照的恢复机制，重建出错批次涉及的图层对象，从而防止引擎呈现“改了一半，另一半报错”的脏状态。

当前尚未提供：

- Layer 粒度的原子事务（源/图层增删暂无自动备忘录恢复）。 
- 批量期间的状态隔离多重试图等高级特性。

## 4. 对外新增信号

本轮为 Map 增加了以下对外信号：

- sourceAdded
- sourceUpdated
- sourceRemoved
- layerAdded
- layerRemoved
- layerVisibilityChanged
- layerOrderChanged
- layerGroupChanged
- layerBatchCommitted
- layoutPropertyChanged
- paintPropertyChanged
- stylePatchApplied
- styleBatchCommitted

这些信号的来源是 LayerManager 和 StyleManager 内部事件，再由 Map 转发给外部调用方。

能力价值：

- 页面层和插件层不需要主动轮询图层状态。
- 未来可以直接驱动图层树面板、调试面板、状态同步器。
- 为批量操作后的统一刷新提供了清晰的事件节点。

## 5. 内部结构变化

### 5.1 新增 LayerManager

本轮新增了 LayerManager 作为内部图层管理器。

职责范围包括：

- source 的添加、更新、移除。
- layer 的添加、移除、显隐、排序。
- 图层分组元数据维护。
- 图层元数据快照生成。
- 批量更新状态管理。
- 图层与数据源事件通知。

这一步的意义在于：

- 让图层与数据源相关逻辑不再继续集中堆积到 Map 中。
- 为后续继续拆分 StyleRuntime 和 EditRuntime 建立可复用结构。

### 5.2 Map 开始转为轻量外观

本轮没有直接删除 Map 现有接口，而是让 Map 的部分 source/layer 操作改为委托给 LayerManager。

这样做的好处是：

- 不打断现有外部 API。
- 不要求调用方一次性全部迁移。
- 可以逐步把职责从大类中抽离，而不是一次性激进重构。

当前已经完成委托的主要接口包括：

委托给 LayerManager 的接口：

- addSource
- sourceExists
- updateSource
- removeSource
- addLayer
- layerExists
- removeLayer
- layerIds
- setLayerVisible
- isLayerVisible
- moveLayer
- setLayerGroup
- groupLayerIds
- setGroupVisible
- removeGroup
- layerMetadata
- layerMetadataSnapshot
- beginLayerBatch
- commitLayerBatch
- isLayerBatchActive

委托给 StyleManager 的接口：

- setLayoutProperty
- setPaintProperty
- setLayoutExpression
- setPaintExpression
- applyStylePatch
- setFilter
- getFilter
- registerLayerStateStyle
- setLayerStateStyleActive
- unregisterLayerStateStyle
- activeLayerStateStyles
- beginStyleBatch
- commitStyleBatch
- isStyleBatchActive
- lastStyleErrorCode
- lastStyleErrorMessage
- styleSnapshot
- stylePropertySnapshot

### 5.3 新增 StyleManager

本轮新增了 StyleManager 作为内部样式管理器。

职责范围包括：

- layout 属性更新。
- paint 属性更新。
- layout/paint 简单表达式入口。
- layer 级状态样式注册、启用与恢复。
- layout/paint 样式补丁提交。
- 图层过滤器（filter）的设置与读取。
- 属性白名单校验与错误状态输出。
- 样式快照与按属性快照读取。
- 样式批量事件合并。
- 样式批量失败时对 filter/layout/paint 相关 layer 快照的统一恢复。
- 样式变更事件通知。
- 图层被删除时自动清理关联的状态样式数据。

这一步的意义在于：

- 让样式更新逻辑开始脱离 Map 大类。
- 为后续表达式入口、状态样式入口、样式快照能力预留独立扩展位置。
- 延续 LayerRuntime 已建立的“Map 外观 + 内部管理器”结构。

## 6. 代码结构约束执行情况

本轮实现遵循了以下约束：

- 函数内代码尽量控制在 100 行以内。

- 采用新的内部类承接职责，而不是继续扩展 Map 单类职责。
- 保持现有 Qt/C++ 命名和组织风格，不引入另一套风格体系。

## 7. 当前边界与限制

虽然 LayerRuntime 已经开始落地，但当前实现仍然有明确边界：

### 7.1 LayerBatch 不是完整事务（StyleBatch具备异常恢复）

当前 `beginLayerBatch` / `commitLayerBatch` 只解决“事件合并”防抖问题，没有解决图层实例增删失败时的“回滚”问题。

这意味着：

- 虽说 `StyleManager` 下已经具备了回滚容错，但如果 `addLayer` 过程图层数据格式受损失败，未对之前批次里的数据源或上一项图层做销毁恢复。

### 7.2 分组目前是运行时元数据

groupId 当前只保存在引擎内部元数据中，不是底层样式规范原生字段。

这意味着：

- 分组信息适合做运行时控制和 UI 展示。
- 但它不会自动体现在底层 style JSON 中。

### 7.3 visibility 修改统一走 setLayerVisible

图层可见性修改只允许通过 setLayerVisible 进行。

StyleManager 的白名单明确拒绝 visibility 属性，调用 setLayoutProperty(layerId, "visibility", ...) 会返回 StyleUnsupportedPropertyError。

这样做的原因是：

- setLayerVisible 走 LayerManager，会触发 layerVisibilityChanged 信号并同步元数据。
- 如果同时允许 StyleManager 修改 visibility，两条路径的事件语义完全不同，会导致内部状态不一致。

### 7.4 图层删除会联动清理 StyleManager 状态数据

当通过 removeLayer 删除图层时，StyleManager 会自动清理该图层关联的状态样式数据。

联动清理的内容包括：

- 已注册的状态样式（m_registeredStateStyles）。
- 当前激活的状态列表（m_activeStateStyles）。
- 基线属性快照（m_trackedStateProperties）。

这是通过 LayerManager 的 layerRemoved 信号连接到 StyleManager 的 onLayerRemoved 槽实现的。

### 7.5 addCustomLayer 已纳入 LayerManager 感知

通过 addCustomLayer 添加的自定义渲染图层现在会触发 layerAdded 和 layerOrderChanged 信号，并同步到图层元数据中。

这意味着 layerIds() 和 layerMetadataSnapshot() 结果中会包含自定义图层。

### 7.6 source/layer 委托接口返回值已修正

addSource、updateSource、removeSource、addLayer、removeLayer、setLayerVisible 的 Map 层返回类型已从 void 改为 bool，与 LayerManager 内部返回值保持一致。

sourceExists 和 layerExists 已添加 const 修饰。

### 7.7 setFilter/getFilter 已纳入 StyleManager

setFilter 和 getFilter 不再直接访问底层样式对象，而是委托给 StyleManager。

filter 修改不走白名单校验（filter 不是 paint/layout 属性），但走图层存在性校验和错误状态设置。
- filter 在 StyleBatch 中已纳入快照恢复范围，但当前仍未提供独立的 filter 变更信号。

### 7.8 layer state style 基线恢复边界已收紧

当前 layer state style 只允许注册到“已有且为 constant 类型”的基线属性上。

这样做的原因是：

- 表达式型或其他非 constant 的属性虽然可以被读取到 kind，但当前首轮状态样式恢复逻辑并不能稳定还原其原始结构。
- 在 P0 阶段先收紧为 constant 基线，可以避免 state style 退出后残留错误样式。

### 7.9 MapPrivate::setProperty 已清理

MapPrivate 中遗留的 setProperty 方法已删除，该方法在委托到 StyleManager 后已无调用点。

### 7.10 还没有完整构建验证记录

当前已做的是编辑器级错误检查，未完成完整工程构建验证。

原因是当前工作区还没有通过 CMake Tools 返回有效 build targets，因此当前只能确认：

- 新增代码在编辑器错误检查层面没有报错。
- 还不能确认全工程编译链路已经完整通过。

### 7.11 StyleRuntime 仍处于第一步

当前 StyleManager 已经把最常用的属性更新路径和样式补丁入口收拢起来，但还没有完成完整 StyleRuntime。

这意味着：

- 目前已经覆盖 layout、paint、简单 expression、layer 状态样式和 patch 提交五类入口。
- 已经形成按图层类型的首轮样式白名单，以及样式快照读取接口。
- 批量更新当前只解决事件合并，不解决样式回滚。
- 当前状态样式仍是 layer 级补丁，不是完整 feature-state 运行时模型。
- 更高层表达式 builder 仍未建立。

## 8. 下一步建议

基于当前已落地的结果，下一步建议有两个方向：

### 方向一：继续完善 LayerRuntime

优先补齐：

- 批量显隐接口。
- 按组批量控制接口。
- 更完整的批量事务语义。
- 图层事件和元数据的统一测试。

### 方向二：继续完善 StyleRuntime

优先拆分：

- 状态样式入口
- 更高层表达式 builder
- 差量提交缓存
- 样式白名单测试和真实调用迁移
- 状态样式和表达式入口的真实业务迁移

如果按 P0 的依赖顺序推进，建议下一步继续补齐 StyleRuntime，再进入 EditRuntime，这样可以先把编辑所依赖的样式入口稳定下来。

## 9. 结论

本轮改动的本质不是“增加了几个工具函数”，而是完成了 LayerRuntime 的第一阶段落地，并启动了 StyleRuntime 的第一步拆分：

- 图层与数据源管理开始脱离 Map 大类。
- 图层运行时控制能力开始具备统一入口。
- 图层元数据与事件通道已经打通。
- 样式属性更新与样式补丁提交通道开始脱离 Map 大类。
- 样式白名单、错误状态、样式快照和表达式透传入口已经建立。
- layer 级状态样式入口和基线恢复能力已经建立。
- 样式批量事件通道已经建立。
- 后续继续拆分 StyleRuntime 和 EditRuntime 已经有了结构基础。

当前引擎仍然没有完成 P0 全部目标，但已经从“图层逻辑分散在大类中”迈向“图层逻辑具备独立运行时管理器”的阶段。
