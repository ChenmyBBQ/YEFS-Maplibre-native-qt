# YEFS 地图引擎功能梳理

## 1. 文档目的

本文档用于从地图引擎平台建设视角，梳理一个合格且完善的地图引擎应具备的能力，并结合 YEFS 当前基于 Qt6/QML + MapLibre Qt 封装的现状，形成阶段性能力盘点。

本文档主要回答以下问题：

- 一个可用于桌面 GIS 产品的地图引擎，基础能力应该覆盖哪些方面。
- 当前 YEFS 已经具备了哪些能力。
- 哪些能力是近期最应优先补齐的。
- 哪些能力属于后续可逐步演进的高级能力。

## 2. 参考基线

本次梳理参考的“成熟平台”不是某一个单独产品，而是综合以下平台的常见能力模型：

- ArcGIS Runtime 一类商用 GIS SDK。
- Cesium / osgEarth 一类三维地理引擎。
- Mapbox / MapLibre 一类矢量样式地图引擎。
- Qt Location / QMapLibre 一类 Qt 生态地图接入层。

这里的“成熟”不特指商业闭源，而是指具备以下特点：

- 图层与数据源抽象清晰。
- 样式、交互、查询、渲染与编辑能力分层明确。
- 能支撑复杂业务场景，而不是只完成地图显示。
- 具备性能治理、调试诊断与可持续扩展能力。

## 3. 面向模块化研发的能力总表

如果只是做能力罗列，文档很容易停留在“知道要做什么”，但无法直接转成模块研发任务。

因此这里对原始能力域做两类处理：

- 把重复度高、边界接近的能力合并为统一模块，例如把“地图显示基础、相机控制、坐标转换”合并为“地图视图与空间参考”。
- 把过于宽泛的大项继续拆到二级能力，使其可以直接映射为接口、控制器、服务或插件研发包。

合并后的模块划分如下。

| 一级模块 | 合并范围 | 二级能力拆分 | 合格引擎应具备的关键能力 | 建议研发产物 |

### 为什么这样拆分

这个拆分方式更适合后续研发，原因在于：

- 每个一级模块都对应明确职责边界，避免把“地图功能”继续做成单体类或万能单例。
- 每个二级能力都可以进一步落到一个接口组、控制器组或服务组，而不是停留在产品术语层。
- 相近能力已被合并，减少了“图层、样式、标注、查询”之间重复建模的风险。

## 4. 当前 YEFS 能力映射

结合当前代码基线，YEFS 在上述模块上的能力状态可整理如下。

| 一级模块 | 当前状态 | 已具备能力 | 当前缺口 |
| --- | --- | --- | --- |
| 地图视图与空间参考 | 已具备 | 已有地图主视图装载、样式切换、中心点/缩放/俯仰/朝向/flyTo、屏幕与地理坐标互转 | 视口状态标准化、统一的地图状态快照接口仍偏弱 |
| 数据与图层管理 | 部分具备 | 已支持 GeoJSON 图层添加、删除、可见性控制、数据更新 | source 类型仍偏少，图层树、分组编排、元数据模型不足 |
| 样式与表达系统 | 部分具备 | 已支持 fill-color、fill-opacity、line-color、line-width、line-dasharray 等属性 | paint/layout 对外 API 不完整，表达式与状态样式能力薄弱 |
| 交互与查询 | 部分具备 | 已具备地图点击、悬停坐标上报、消息总线广播 | 业务对象命中、要素查询、过滤查询、批量命中测试仍不足 |
| 编辑与绘制链路 | 部分具备 | 空域插件已实现绘制、预览、确认提交的主链路 | 节点编辑、拖拽修改、撤销重做、统一编辑会话模型缺失 |
| 渲染性能治理 | 部分具备 | 已有高频 annotation 预览旁路，已有局部优化意识 | 平台级差量更新、批量提交、统一节流策略尚未沉淀 |
| 标注与专题表达 | 缺口明显 | 当前具备少量基础样式表达能力 | label/symbol、注记、聚合、专题渲染尚未成体系 |
| 三维与地形能力 | 基本缺失 | 当前仅具备俯仰等基础视角能力 | terrain、DEM、挤出、三维对象表达尚未建立 |
| 空间工具与分析挂接 | 基本缺失 | 坐标转换已具备，可为工具层提供基础支撑 | 测量、缓冲、范围查询、分析结果叠加尚未形成模块 |
| 离线与资源管理 | 基本缺失 | 当前主要围绕在线样式与业务图层运行 | 离线包、缓存控制、资源预载、弱网策略未成体系 |
| 调试诊断与运维 | 缺口明显 | 已有运行日志和消息总线日志基础 | 图层树检查、source/layer 状态检查、渲染统计、诊断面板缺失 |
| 平台集成与扩展 | 已具备基础 | 已有 IMapEngine 抽象、MessageBus、插件接入能力，空域插件已落地 | 地图能力权限边界、插件侧标准接口与模块协议仍不够规范 |

### 当前能力评价

当前 YEFS 地图引擎已经达到“可承载主业务页面与插件联动”的水平，属于可用的业务型地图内核，但还未达到“成熟地图平台层”的水平。

其主要特点是：

- 地图视图、基础图层、基础样式和插件挂接已经打通。
- 空域插件证明了当前引擎具备承载真实业务闭环的能力。
- 主要短板已经从“能不能显示”转移到“能否模块化、可维护地持续扩展”。

## 5. 优先补充的能力

这部分不再只列“能力点”，而是直接列为可落地的研发模块包。这样后续任务拆分、排期和接口设计会更直观。

| 优先级 | 建议研发模块包 | 对应一级模块 | 首批应交付的内容 | 为什么优先 |
| --- | --- | --- | --- | --- |
| P0 | LayerRuntime | 数据与图层管理、渲染性能治理 | source 更新、layer 更新、图层元数据、图层分组、可见性与排序统一接口 | 这是后续所有样式、查询、编辑能力的底座 |
| P0 | StyleRuntime | 样式与表达系统、渲染性能治理 | paint/layout 分层更新、样式差量提交、状态样式入口、表达式接口封装 | 先解决整层刷新与样式拼装分散问题 |
| P0 | EditRuntime | 编辑与绘制链路 | 预览层与正式层统一策略、编辑会话模型、绘制提交协议、基础撤销重做 | 编辑链路是当前最贴近业务的高频场景 |
| P1 | QueryRuntime | 交互与查询 | 要素拾取、命中测试、queryRenderedFeatures 封装、业务对象反查 | 没有查询能力，地图就难以成为业务交互平台 |
| P1 | LabelRuntime | 样式与表达系统、标注与专题表达 | label/symbol 图层、文本样式、图标资源、名称注记基础能力 | 后续对象名称、空域标注、态势注记都依赖它 |
| P1 | DiagnosticRuntime | 调试诊断与运维 | 图层树检查、source/layer 状态、渲染统计、事件追踪面板 | 平台一旦复杂，诊断能力决定维护成本 |
| P1 | ObjectBindingRuntime | 数据与图层管理、交互与查询 | source-layer-feature 与业务对象映射表、对象状态同步 | 这是支撑查询、编辑、诊断闭环的基础元数据层 |
| P2 | OfflineRuntime | 离线与资源管理 | 瓦片缓存、资源预载、离线包框架、资源版本管理 | 桌面 GIS 后续常见刚需，但不必先于核心交互链路 |
| P2 | SpatialToolRuntime | 空间工具与分析挂接 | 测量、缓冲、范围分析挂接接口、分析结果覆盖物 | 能显著提升产品工具属性，但依赖前置图层与交互能力 |
| P2 | Scene3DRuntime | 三维与地形能力 | terrain、DEM、挤出、三维对象表达基础框架 | 价值高，但应放在二维平台能力稳定之后 |

### 优先补充能力的建议建设顺序

| 阶段 | 建议内容 | 目标 |
| --- | --- | --- |
| 第一阶段 | 完成 LayerRuntime 与 StyleRuntime，明确 source、layer、paint、layout 四层更新接口 | 先解决“整体重绘”和接口分散问题 |
| 第二阶段 | 完成 EditRuntime 与 QueryRuntime，统一绘制、预览、提交、查询链路 | 让地图进入“可编辑、可命中、可反馈”的平台阶段 |

### 把能力文档转成执行任务的规则

当前文档前半部分解决的是“应该建设什么”，但研发真正卡住的往往是“这一轮到底做多少，做到什么程度算完成”。

因此后续所有模块研发都建议强制补齐以下四类信息，否则不要直接进入开发排期：

- 本轮目标：这一轮只解决什么问题，必须用一句话说清。
- 本轮范围：明确列出本轮纳入的接口、场景和页面，不允许只写“支持某能力”。
- 非目标：明确列出这一轮不做什么，避免研发过程中范围不断膨胀。
- 验收标准：必须是可验证的结果，而不是“能力已支持”“基本可用”这类主观表述。

建议把每个模块都按下面的任务卡片模板落地，而不是继续用泛化能力点直接派工：

| 字段 | 要求 |
| --- | --- |
| 模块名 | 必须对应唯一模块，不要一个任务跨多个一级模块 |
| 本轮目标 | 只描述一轮迭代目标，不描述长期愿景 |
| 本轮范围 | 列出具体接口、具体对象类型、具体页面或插件场景 |
| 非目标 | 列出至少 3 项明确不做的内容 |
| 前置依赖 | 说明依赖哪个模块、哪种数据结构或哪条链路先稳定 |
| 交付物 | 至少包含代码、接口文档、示例或测试之一 |
| 验收标准 | 必须能通过 demo、测试或界面行为直接验证 |

### 首批模块的执行定义

下面这张表不是长期蓝图，而是建议直接用于拆分近期任务的首批执行定义。它的目的不是追求完整，而是消除“描述正确但无法开工”的问题。

| 模块 | 本轮必须交付 | 本轮明确不做 | 最低验收标准 |
| --- | --- | --- | --- |
| LayerRuntime | 统一 addSource、addLayer、removeLayer、setLayerVisible、moveLayer；建立图层元数据结构；支持图层分组与顺序调整 | 不做完整图层树面板；不做所有 source 类型一次补齐；不做跨页面复杂权限体系 | 同一业务页面内可通过统一接口完成图层增删改查；图层顺序与显隐不再散落在页面代码中；至少有一个真实业务插件完成迁移 |
| StyleRuntime | 统一 setPaintProperty、setLayoutProperty、applyStylePatch；支持 fill、line、symbol 三类基础样式入口；支持样式差量提交 | 不做全量样式规范封装；不做复杂表达式编辑器；不做全部 feature-state 能力 | 样式修改不再依赖整层重建；基础样式修改能覆盖现有主页面和空域插件；至少有一组差量更新测试或示例 |
| QueryRuntime | 封装 hitTest、queryRenderedFeatures、按图层过滤查询；返回 feature 与业务对象的基础关联结果 | 不做复杂空间分析；不做跨引擎统一查询抽象；不做批量空间统计 | 点击地图可稳定返回命中要素；能按图层或图层组限制查询范围；业务层不再直接调用底层零散查询接口 |
| EditRuntime | 建立 beginEditSession、updatePreview、commitEdit、cancelEdit；统一预览层与正式层切换；提供基础撤销重做 | 不做所有几何类型一次补齐；不做高级拓扑编辑；不做多人协同编辑 | 点线面至少一类核心业务对象完成统一编辑链路；预览与提交走同一套会话模型；取消编辑后临时要素可完全清理 |
| ObjectBindingRuntime | 建立 source-layer-feature 到业务对象 ID 的映射；支持对象状态同步；支持通过查询结果反查业务对象 | 不做完整业务仓储；不做复杂缓存持久化；不做跨进程对象同步 | 查询结果可稳定定位到业务对象；对象删除或更新后映射关系可同步更新；编辑、查询、诊断三处不再各自维护一份对象关联逻辑 |

### P0 问题收敛与功能拆分

如果当前目标是“先解决 P0”，那么执行范围必须严格收敛为以下三个模块，而不是继续把 P1、P2 能力混入当前迭代。

| P0 模块 | 当前要解决的核心问题 | 本轮目标 |
| --- | --- | --- |
| LayerRuntime | 图层与数据源操作分散在页面、插件和临时逻辑中，缺少统一入口 | 建立统一的 source/layer 管理接口与元数据模型 |
| StyleRuntime | 样式修改分散、整层重建偏多、缺少统一样式入口 | 建立统一的 paint/layout 更新与差量样式提交通道 |
| EditRuntime | 预览、编辑、提交、取消逻辑分散，缺少会话模型 | 建立统一编辑会话与预览/提交主链路 |

当前阶段不把 QueryRuntime、LabelRuntime、ObjectBindingRuntime 纳入 P0，是为了避免“底座还没稳定，就开始补上层功能”，导致实现边界再次混乱。

### P0 总体执行顺序

| 顺序 | 模块 | 先做它的原因 | 直接依赖关系 |
| --- | --- | --- | --- |
| 1 | LayerRuntime | 样式、编辑、查询都依赖统一图层与 source 管理能力 | 无 |
| 2 | StyleRuntime | 图层统一后，样式更新才能从页面逻辑中抽离 | 依赖 LayerRuntime 提供稳定 layer 标识与元数据 |
| 3 | EditRuntime | 编辑链路最终要依赖统一图层与样式接口来落预览和提交 | 依赖 LayerRuntime 与 StyleRuntime |

### P0-1 LayerRuntime 功能拆分

LayerRuntime 的目标不是“做一个图层面板”，而是先把图层控制逻辑从页面和插件代码中抽出来，形成唯一可信入口。

| 一级功能 | 细功能点 | 首轮必须交付 |
| --- | --- | --- |
| Source 注册与移除 | addSource；removeSource；source ID 唯一性检查；重复注册保护；source 存活状态记录 | 必须 |

### P0-2 StyleRuntime 功能拆分

StyleRuntime 的目标不是一次性封装全部样式规范，而是先统一当前最常用、最容易失控的样式更新路径。

当前已开始落地的首轮范围如下：

- 已新增内部 StyleManager，作为样式更新统一入口。
- 已把 setLayoutProperty 和 setPaintProperty 从 Map 直接处理改为委托 StyleManager。
- 已新增 setLayoutExpression 和 setPaintExpression，作为简单表达式透传入口。
- 已新增 StyleExpression 辅助 builder，用于构造 get、feature-state、boolean feature-state、eq、case 等常用表达式片段。
- 已新增 applyStylePatch，用于按图层提交 layout/paint 补丁。
- 已新增 layer 级状态样式入口，可注册状态补丁、启用状态补丁并恢复基线样式。
- 已新增 beginStyleBatch、commitStyleBatch、isStyleBatchActive，用于样式批量事件合并。
- 已新增 lastStyleErrorCode 和 lastStyleErrorMessage，用于返回清晰错误状态。
- 已新增 styleSnapshot 和 stylePropertySnapshot，用于样式回显与调试读取。
- 已为首轮 StyleRuntime 加上按图层类型的属性白名单约束，当前范围收敛为 fill、line、symbol、circle 四类高频层。
- 已新增 layoutPropertyChanged、paintPropertyChanged、stylePatchApplied、styleBatchCommitted 信号，用于外部感知样式变更。

当前仍未纳入首轮落地的内容如下：

- 更高层表达式 builder 还未补齐，目前只覆盖常用片段 builder。
- 状态样式当前是 layer 级状态补丁，不是完整 feature-state 运行时模型。
- 批量更新当前仍然是事件合并，不是完整事务回滚。

| 一级功能 | 细功能点 | 首轮必须交付 |
| --- | --- | --- |
| Paint 属性更新 | setPaintProperty；按 layerId 更新；属性名校验；属性值类型校验；错误返回 | 必须 |
| Layout 属性更新 | setLayoutProperty；layout 属性校验；非法属性拦截；更新后状态同步 | 必须 |
| 基础图层样式入口 | fill 样式入口；line 样式入口；symbol 样式入口；统一参数结构 | 必须 |
| 样式补丁提交 | applyStylePatch；一次提交多个 paint/layout 变更；按图层批量更新 | 必须 |
| 差量样式提交 | 仅提交变更属性；避免整层重建；避免页面层直接拼装底层样式调用 | 必须 |
| 样式能力白名单 | 明确首轮支持哪些 paint/layout 属性；不支持属性返回清晰错误 | 必须 |
| 表达式入口封装 | 常量值；简单表达式透传；统一表达式参数入口 | 建议首轮纳入 |
| 状态样式入口 | 为后续选中、高亮、告警预留状态样式入口，但不承诺完整 feature-state | 建议首轮纳入 |
| 样式快照读取 | 读取当前图层样式；导出当前已应用的样式补丁；支持调试与回显 | 建议首轮纳入 |

#### StyleRuntime 进一步拆到可派工的细功能点

| 功能项 | 需要继续拆分的细点 |

### P0-3 EditRuntime 功能拆分

EditRuntime 的目标不是把所有编辑能力一次补齐，而是先打通统一编辑会话，让“开始编辑、更新预览、提交、取消、撤销”成为一条稳定主链路。

| 一级功能 | 细功能点 | 首轮必须交付 |
| --- | --- | --- |
| 编辑会话管理 | beginEditSession；sessionId；当前编辑对象记录；结束会话；并发会话保护 | 必须 |
| 预览层管理 | 创建预览图层；更新预览几何；销毁预览图层；预览层与正式层隔离 | 必须 |
| 编辑输入更新 | updatePreview；追加点；修改点；拖拽更新；按编辑动作更新几何 | 必须 |
| 提交与取消 | commitEdit；cancelEdit；提交前校验；取消后临时状态清理 | 必须 |
| 撤销重做 | undo；redo；操作栈记录；清空重做栈；会话结束时释放操作栈 | 必须 |
| 几何类型边界 | 首轮支持的几何类型清单；点、线、面至少选定一类核心对象先落地 | 必须 |

### P0 拆分后的直接派工方式

为了避免再次回到“知道要做什么，但不知道怎么排任务”的状态，P0 建议直接按下面的层级拆任务：

1. 先按模块拆：LayerRuntime、StyleRuntime、EditRuntime。
2. 再按一级功能拆：例如 LayerRuntime 先拆 Source 注册、Layer 显隐、图层排序、图层分组。
3. 最后按细功能点拆成开发任务：例如 addLayer 的参数结构、插入位置规则、重复添加保护分别作为独立实现点。

只有拆到“一个开发任务只解决一个细功能点”这个粒度，排期、分工、联调和验收才会稳定。

### 用什么表述才算不模糊

后续文档中建议避免继续使用下面这类无法指导执行的表述：

- “支持图层控制”
- “完善查询能力”
- “补齐标注能力”
- “增强三维能力”

这些说法的问题在于它们只能表达方向，不能表达工作边界。

建议改写成下面这种可以直接用于排期和派工的表达：

- “在 LayerRuntime 中统一图层显隐、排序、分组接口，首轮只覆盖 GeoJSON 图层和业务动态图层。”
- “在 QueryRuntime 中提供按图层过滤的点击命中查询，并返回业务对象 ID，不包含框选与统计分析。”
- “在 LabelRuntime 中首轮只支持名称注记、图标资源和最小缩放级别控制，不包含碰撞编辑器。”
- “在 Scene3DRuntime 中首轮只验证挤出图层接入，不承诺 terrain 与三维模型完整能力。”

### 文档如何指导下一步执行

如果要让这份文档真正指导下一步研发，建议按以下顺序使用，而不是把它仅作为能力盘点材料：

1. 先从“优先补充的能力”表中只选 1 到 2 个模块进入当前迭代。
2. 再用上面的“任务卡片模板”给每个模块补齐本轮目标、范围、非目标和验收标准。
3. 最后再拆成开发任务、接口设计任务、测试任务，而不是直接按功能名分工。

也就是说，这份文档本身不应再承担“任务单”的职责，而应承担“模块优先级与边界定义”的职责。真正进入研发排期时，必须再下沉一层，形成模块任务卡片，否则执行过程一定会继续发散。

## 7. 建议的模块化研发边界

如果后续要按功能表推进模块研发，建议按下列边界组织代码与任务，而不是继续围绕单个页面或单个插件零散补丁式演进。

| 建议模块 | 职责边界 | 不应承担的职责 | 典型对外接口 |
| --- | --- | --- | --- |
| EngineCore | 地图视图绑定、引擎生命周期、统一入口 | 业务绘制规则、业务对象存储 | 初始化、地图实例绑定、引擎状态查询 |
| LayerRuntime | source/layer 注册、排序、分组、可见性、元数据 | 业务页面交互细节 | addSource、addLayer、updateLayer、removeLayer |
| StyleRuntime | paint/layout/表达式/状态样式更新 | 图层生命周期管理 | setPaintProperty、setLayoutProperty、applyStylePatch |
| QueryRuntime | 命中测试、要素查询、对象映射查询 | 持久化、业务编辑流程 | hitTest、queryFeatures、resolveBusinessObject |
| EditRuntime | 绘制、预览、编辑会话、提交、撤销重做 | 具体业务对象存储策略 | beginEditSession、updatePreview、commitEdit |
| LabelRuntime | label/symbol、注记、图标资源、聚合表达 | 通用图层生命周期 | addLabelLayer、setTextStyle、setIconStyle |
| DiagnosticRuntime | 图层树、source 状态、性能统计、事件追踪 | 业务对象编辑规则 | dumpLayerTree、collectStats、traceEvent |
| OfflineRuntime | 缓存、离线包、资源版本、预加载 | 在线业务对象状态机 | preloadResources、importOfflinePack、switchOffline |
| SpatialToolRuntime | 测量、缓冲、范围分析、分析结果覆盖物 | 地图底层渲染实现 | measureDistance、bufferGeometry、runAnalysis |
| Scene3DRuntime | terrain、DEM、挤出、三维对象 | 二维业务通用查询逻辑 | enableTerrain、addExtrusionLayer、setSceneMode |

这张表的意义在于明确“模块职责”，防止后续把查询逻辑塞进样式层、把编辑状态塞进地图页面、把离线逻辑塞进插件层，导致边界继续混乱。

## 8. 当前平台定位结论

基于当前代码现状，可以把 YEFS 的地图引擎能力定位为：

| 维度 | 当前结论 |
| --- | --- |
| 地图显示能力 | 已达到可用水平 |
| 业务承载能力 | 已能支撑主页面和空域插件等现有业务 |
| 平台抽象成熟度 | 仍偏基础，尚未达到成熟平台层 |
| 性能治理能力 | 已开始有优化思路，但平台级能力不足 |
| 高级 GIS 能力 | 仍有明显缺口 |

简而言之：

- 当前 YEFS 地图引擎已经不是“演示型”能力，而是“业务可用型”能力。
- 但它还不是一个“完善的地图平台层”。
- 下一步最值得投入的方向，不是继续堆叠零散业务逻辑，而是把图层、样式、查询、编辑、诊断这些平台能力拆成独立研发模块。

## 9. 建议的后续路线

如果后续要持续建设地图引擎层，建议按如下路线推进：

| 路线阶段 | 建议目标 | 预期结果 |
| --- | --- | --- |
| 近期 | 完成 LayerRuntime、StyleRuntime、EditRuntime 的基础骨架 | 地图更新从“整层更新”转为“差量更新”，编辑主链路统一 |
| 近期 | 完成 QueryRuntime 与 LabelRuntime 基础能力 | 地图具备命中查询、名称注记、对象反馈的基本平台属性 |
| 中期 | 完成 ObjectBindingRuntime 与 DiagnosticRuntime | 形成对象映射、状态跟踪、诊断排查的可维护平台层 |
| 中期 | 完成 SpatialToolRuntime 的基础测量与分析挂接能力 | 地图从展示内核升级为工具内核 |
| 远期 | 按业务需求补齐 OfflineRuntime 与 Scene3DRuntime | 支撑离线外场、2.5D/3D 与更复杂的场景表达 |

这套路线能保证平台能力建设与当前业务演进方向一致，也更符合 YEFS 作为 Qt6/QML/C++ 桌面 GIS 应用的长期架构约束。
